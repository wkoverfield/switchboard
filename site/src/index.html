[ 02 ] · the loop

Three commands to a governed agent

Start in a repo where your agents already work. Switchboard meets your existing MCP config where it is.

01 / see it

scan, then import

$ switchboard scan
high claude:github
  token in plaintext .mcp.json
$ switchboard import --write
  token → OS keychain
  backup + exact rollback kept

Every server and token your agents can already touch, including the routes that bypass everything.

02 / scope it

grant a pass

$ switchboard grant --for 4h
SWITCHBOARD · PASS GRANTED
  → github_ci_*
  everything else denied
  expires in 4h

Named tools reachable, everything else denied, gone on its own. Revoke early any time.

[ 03 ] · control

The pass is the perimeter.

Everything routed through Switchboard obeys it. Full boundaries in the threat model.

Denied before it leaves the machine.

Out-of-scope calls die inside Switchboard; the provider never sees them. Routes that go around the pass, like raw shell or a direct MCP entry, are exactly what switchboard scan finds and reports.

Risky calls wait for a human.

Approval gates hold a call until you decide, in your client or from the CLI. Provider templates ship sane gates for GitHub, Vercel, Stripe, and Supabase.

Access that ends itself.

Passes expire on their own; revoke ends access at the next routed call. No pass means configured profiles run without tool policy, so keep one live; default-deny is on the roadmap.

Alpha, with stable contracts.

Local-first and moving fast. Passes and audit entries are versioned JSON, so scripts and harnesses can build on them today.

[ 04 ] · security

The security work is written down

Threat model, audit log design, and secrets handling, documented with code references.

threat-model.md Threat model STRIDE analysis of every surface: what enforcement binds, what it cannot, revocation semantics, and the accepted risks. Read it.
audit verify Tamper-evident audit log Every routed call, denial, and approval decision is hash-chained JSONL on your machine. switchboard audit verify reports the exact line where a chain breaks. How it works.
secretRef Secrets stay in the OS keychain Config carries named refs, never values. Weaker backends require an explicit unsafe opt-in. Values are injected only into the upstream that declared them, at spawn, and redacted everywhere else.
child ⊆ parent Delegation that cannot widen A lead agent can hand narrower passes to subagents; children are validated as strict subsets of the parent, and parent deny lists are inherited.
